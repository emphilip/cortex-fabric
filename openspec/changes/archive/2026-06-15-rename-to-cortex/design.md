# Design — rename Hive Mind → Cortex

## Context

~1,008 references to the name across ~126 files in four casings: `hive-mind` (kebab), `hive_mind` (snake), `HIVE_MIND` (env), and the `@hive-mind` npm scope. The rename is mechanical but spans package manifests, lockfiles, source, SQL, compose, OTel/Prom, tests, and docs — so order and verification matter to avoid a half-renamed, non-booting stack.

## Decisions

### D1 — Short internal name `cortex`, descriptive repo `cortex-context-fabric`
Internal identifiers use the short, ergonomic `cortex` (snake `cortex`, kebab `cortex`, env `CORTEX`, scope `@cortex`). Only the GitHub repository and top-level directory use the descriptive `cortex-context-fabric`. Rationale: `@cortex-context-fabric/admin-ui`, `CORTEX_CONTEXT_FABRIC__PIPELINE__URL`, and `cortex_context_fabric_pipeline` would be painfully verbose and churn-prone; the repo name is the only place the full descriptor earns its keep (discoverability/branding).

**Alternative considered:** full `cortex_context_fabric` everywhere — rejected for verbosity. If you prefer the long form internally, this is the one decision to flip before we start.

### D2 — Casing map (apply per token, not blanket)
- `@hive-mind/` → `@cortex/`
- `hive-mind` (kebab, non-scope) → `cortex` for image names/config ids; the repo dir/remote → `cortex-context-fabric`
- `hive_mind` (snake: Python packages, metric prefixes, DB schema) → `cortex`
- `HIVE_MIND` (env prefix, config section keys) → `CORTEX`
- Prose "Hive Mind" → "Cortex"

A blanket find/replace is unsafe because kebab `hive-mind` maps to two targets (`cortex` vs `cortex-context-fabric`). We replace the snake/env/scope forms globally, but handle bare kebab `hive-mind` deliberately (repo/dir vs image/id).

### D3 — Postgres schema: fresh dev volume, not in-place ALTER
The schema `hive_mind` (with append-only triggers on `hive_mind.audit_log` / `hive_mind.graph_audit_log`, plus Apache AGE graph) is created by the numbered init SQL. We rename the schema to `cortex` in the init SQL and recreate the dev volume rather than authoring an `ALTER SCHEMA` migration: this is the dev profile, data is reproducible via `make ingest-git` + smoke, and there is no production database. All `asyncpg` query strings referencing `hive_mind.` move to `cortex.`.

**Risk:** a running stack against the old volume will mismatch. Mitigation: `make down`, drop the postgres volume, `make up-d` re-inits with the new schema, then `bash tests/smoke/run.sh` to repopulate and verify.

### D4 — Verification gates
After the rename, the stack must come up clean and the smoke must pass — that is the real proof a 1,000-string rename didn't break a query, an env lookup, or a service URL. Gates, in order:
1. `grep -rIn -iE "hive[-_ ]?mind"` over tracked files (excluding archived openspec + this change's own prose) returns **zero** unexpected hits.
2. `uv sync --all-packages` + `pnpm install` resolve under the new names.
3. `pnpm --filter @cortex/admin-ui build` + admin-ui tests + Storybook build green.
4. `make down` (remove volume) → `make up-d` → all services healthy.
5. `bash tests/smoke/run.sh` PASSES end-to-end (proves DB schema, env, metrics, service URLs all wired).
6. Secret-scan staged diff; commit in batches; rename GitHub repo; update `origin`.

### D5 — Lockfiles & generated artifacts
`pnpm-lock.yaml` and `uv.lock` embed the workspace package names and must be regenerated (not hand-edited). `node_modules`, `.next`, `storybook-static` are not edited (regenerated by build). Archived OpenSpec changes under `openspec/changes/archive/**` are historical records and are **not** rewritten; their old name is acceptable.

### D6 — GitHub remote rename ordering
Rename the repo on GitHub last (after code is pushed under the new name on the same `main`), then `git remote set-url origin git@github.com:emphilip/cortex-context-fabric.git`. GitHub auto-redirects the old URL, so timing is forgiving. The local working-directory rename (`hive-mind` → `cortex-context-fabric`) is optional and done by the operator since it changes absolute paths in the running shell/compose.

## Open questions

- **D1 short vs long internal name** — proceeding with short `cortex`; flip before start if you disagree.
- **Local directory rename** — leave the on-disk folder as `hive-mind` (only rename repo + identifiers) or also move the directory to `cortex-context-fabric`? Moving it changes the compose project name and your shell paths; recommend doing it manually after the code rename lands.
